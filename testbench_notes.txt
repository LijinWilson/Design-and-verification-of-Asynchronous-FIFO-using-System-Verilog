>> SYSTEM VERILOG
    - Architecture of the SV tb is mentioned inside the asset/system_verilog_tb_architecture.png.
    - Components:
        1) Generator: it generate stimulus, send it to the driver through transaction.
        2) Transaction: It act likes an packet, which is used to send and receive data.
        3) Driver: It will send the data to interface.
        4) Monitor: It is an passive element, it just observe the output from the interface and pass the result to the interface.
        5) ScoreBoard: It will compare the  golden result with the result coming from the interface(DUT).

>> WORK FLOW:
    Test
    ├── Generator
    │   ├── (sends transaction via mailbox)
    │   └── Driver
    │       ├── (drives interface signals)
    │       └── DUT
    │
    └── Monitor
        ├── (reads interface signals)
        ├── (sends observed transaction via mailbox)
        └── Scoreboard
            ├── (computes expected)
            └── (compares with DUT result)


>> INTERFACE.SV 
    -> We are using parametrized interface?
        - The interface is parameterized mainly for reusability and scalability. Instead of creating separate interfaces for 8-bit, 16-bit, or 32-bit FIFOs,
            a single interface can support any data width by changing only the parameter value. This also improves maintainability,
            avoids port-width mismatches between the DUT and verification environment, and follows industry-standard IP design practices where modules and interfaces
            are written to be configurable rather than hardcoded.

    -> MODPORT
        - Modport defines the direction of signals from the perspective of the module using the interface.
        
        - Example: Why is FULL an input?

            Who knows when FIFO is full?
            The FIFO.
            FIFO ---> full

            The Testbench only checks it.
            if(vif.full)
            So:
            input full;
        - Example 2: Why is WR_EN an output?

            Who decides when to write?
            The Driver.
            vif.wr_en = 1;

            So the Testbench writes wr_en.
            output wr_en;
    
    -> Easy trick (don't memorize)

        * Just ask yourself one question for every signal:

        * "Who is driving this signal?"

        * If Testbench drives it → OUTPUT in Tb modport.
        * If Testbench only observes it → INPUT in Tb modport.
