>> SYSTEM VERILOG
    - Architecture of the SV tb is mentioned inside the asset/system_verilog_tb_architecture.png.
    - Components:
        1) Generator: it generate stimulus, send it to the driver through transaction.
        2) Transaction: It act likes an packet, which is used to send and receive data.
        3) Driver: It will send the data to interface.
        4) Monitor: It is an passive element, it just observe the output from the interface and pass the result to the interface.
        5) ScoreBoard: It will compare the  golden result with the result coming from the interface(DUT).

>> WORK FLOW:
    Test
    ├── Generator
    │   ├── (sends transaction via mailbox)
    │   └── Driver
    │       ├── (drives interface signals)
    │       └── DUT
    │
    └── Monitor
        ├── (reads interface signals)
        ├── (sends observed transaction via mailbox)
        └── Scoreboard
            ├── (computes expected)
            └── (compares with DUT result)


>> INTERFACE.SV 
    -> We are using parametrized interface?
        - The interface is parameterized mainly for reusability and scalability. Instead of creating separate interfaces for 8-bit, 16-bit, or 32-bit FIFOs,
            a single interface can support any data width by changing only the parameter value. This also improves maintainability,
            avoids port-width mismatches between the DUT and verification environment, and follows industry-standard IP design practices where modules and interfaces
            are written to be configurable rather than hardcoded.

    -> MODPORT
        - Modport defines the direction of signals from the perspective of the module using the interface.
        
        - Example: Why is FULL an input?

            Who knows when FIFO is full?
            The FIFO.
            FIFO ---> full

            The Testbench only checks it.
            if(vif.full)
            So:
            input full;
        - Example 2: Why is WR_EN an output?

            Who decides when to write?
            The Driver.
            vif.wr_en = 1;

            So the Testbench writes wr_en.
            output wr_en;
    
    -> Easy trick (don't memorize)

        * Just ask yourself one question for every signal:

        * "Who is driving this signal?"

        * If Testbench drives it → OUTPUT in Tb modport.
        * If Testbench only observes it → INPUT in Tb modport.

>> TRANSACTION.SV

    Purpose:
        - Represents one FIFO operation.
        - Travels: Generator -> Driver -> Monitor -> Scoreboard.

    Variables:
        - data          : Random data for FIFO.
        - wr_en         : Random write enable.
        - rd_en         : Random read enable.
        - expected_data : Expected value used by Scoreboard.

    Functions:
        - new()      : Initializes variables to known values (avoids X).
        - display()  : Prints transaction for debugging.
        - copy()     : Creates and returns a NEW transaction object with the same values.
                        Used by Monitor to avoid modifying the original transaction.
        - compare()  : Compares current transaction with another transaction.
                    Returns 1 = Match, 0 = Mismatch (used by Scoreboard).

    Keywords:
        - this        : Refers to the current transaction object.
        - transaction : Return type of copy() (returns a transaction object).

    Remember:
        copy()    -> Duplicate Object
        compare() -> Compare Objects

>> GENERATOR.SV
    -> transaction user_queue[$]; 
        - [$]: It is an QUEUE, which is an dynamic array whose size grows and shrinks automatically.
        - Create a queue that can store transaction objects.

    Purpose:
        - Generates transactions and sends them to the Driver through the mailbox.
        - Supports two modes:
            1. Random Mode      -> Generator creates random transactions.
            2. User-defined Mode -> User/Test provides transactions manually.

    Variables:
        - random_mode   : Selects operating mode.
                        1 = Random Mode (default)
                        0 = User-defined Mode

        - user_queue[$] : Queue that stores user-created transaction objects.
                        (Generator DOES NOT create these transactions.)

        - gen2drv       : Mailbox used to send transactions to the Driver.

        - num_transaction : Number of random transactions to generate.

        new(mailbox gen2drv)
        - Connects generator to the mailbox.
        - Sets default mode to Random.

        add_user_transaction(transaction tr)
        - Receives a transaction object from the Test.
        - Stores it at the end of user_queue using push_back().
        - Does NOT create the transaction.

        run()

    Random Mode:
    ------------
        1. Create a new transaction.
        2. Randomize it.
        3. Ensure (wr_en || rd_en) is true.
        4. Display the transaction.
        5. Send it to Driver using gen2drv.put().

    User-defined Mode:
    ------------------
        1. Read each transaction from user_queue.
        2. Display it.
        3. Send it to Driver using gen2drv.put().

    Flow:
    -----
        Random Mode:
        Generator -> Create -> Randomize -> Mailbox -> Driver

        User-defined Mode:
        Test -> user_queue -> Mailbox -> Driver

    Remember:
    ---------
        new()                    -> Connect mailbox.
        add_user_transaction()   -> Store user transaction in queue.
        run()                    -> Send transactions to Driver.


>> DRIVER.SV

    Purpose:
        - Receives transactions from Generator through the mailbox.
        - Converts transaction fields into DUT input signals.
        - Drives the FIFO interface.

    Connections:
        Generator --> Mailbox(gen2drv) --> Driver --> Interface(vif) --> FIFO(DUT)

    Members:
        - virtual fifo_if vif : Handle to the DUT interface (used to drive signals).
        - mailbox gen2drv     : Receives transactions from Generator.

    Constructor:
        - Stores interface and mailbox handles for later use.

    -------------------------------------------------------------------------------
    run() Workflow
    -------------------------------------------------------------------------------

    1. Wait forever for a transaction.
        gen2drv.get(tr);

    2. Check Write Request (tr.wr_en)
        |
        |-- Wait for write clock edge.
        |-- Check FIFO is NOT full and NOT in reset.
        |-- Drive:
                wr_en = 1
                data_in = tr.data
        |-- Next clock:
                wr_en = 0
        |-- Print write message.

    3. Check Read Request (tr.rd_en)
        |
        |-- Wait few read clocks (FIFO flag synchronization).
        |-- Check FIFO is NOT empty and NOT in reset.
        |-- Drive:
                rd_en = 1
        |-- Next clock:
                rd_en = 0
        |-- Print read message.

    -------------------------------------------------------------------------------
    Why these checks?
    -------------------------------------------------------------------------------

    !vif.full
        -> Prevent writing into a full FIFO (avoids overflow).

    !vif.empty
        -> Prevent reading from an empty FIFO (avoids underflow).

    !vif.wr_rst / !vif.rd_rst
        -> Do not perform operations while FIFO is in reset.

    @(posedge vif.wr_clk)
        -> Write only on the write clock edge.

    @(posedge vif.rd_clk)
        -> Read only on the read clock edge.

    repeat(3)
        -> Gives asynchronous FIFO time to synchronize the 'empty' flag before reading.

    Remember:
        - Driver NEVER creates transactions.
        - Driver NEVER checks data correctness.
        - Driver ONLY drives DUT signals based on received transactions.
